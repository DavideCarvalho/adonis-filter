`@agora/filter-client` is the front-end half of the pair. It builds the exact wire format `@adonis-agora/filter` parses ‚Äî `filter[field][op]=value`, `sort=-createdAt`, `page`/`size` ‚Äî through a fluent API, with zero runtime dependencies (it runs in the browser and Node). It also doubles as a reactive store for React/Vue/Svelte and has an optional TanStack Table adapter.

```bash
npm install @agora/filter-client
```

---

## The builder

`filterQuery()` returns a fresh `FilterQueryBuilder`. Chain conditions, then terminate with `toQueryString()`, `build()`, or `toFlatObject()`.

```ts
import { filterQuery } from '@agora/filter-client'

const qs = filterQuery()
  .equals('status', 'active')
  .contains('name', 'fleet')
  .gte('age', 18)
  .sort('createdAt', 'desc')
  .page(1, 25)
  .toQueryString()
// ‚Üí filter[status]=active&filter[name][contains]=fleet&filter[age][gte]=18&sort=-createdAt&page=1&size=25

await fetch(`/users?${qs}`)
```

### `where` ‚Äî the core method

`where` is overloaded to cover every operator shape, and **replaces** any existing filter for the same field (the natural mode for a UI where a control replaces its previous selection):

```ts
filterQuery().where('age', 'gte', 18)        // explicit operator
filterQuery().where('status', ['A', 'B'])    // array ‚Üí auto `in`
filterQuery().where('name', 'Al')            // bare value ‚Üí auto `equals`
filterQuery().where('deletedAt', 'isNull')   // unary operator, no value
```

The value is validated against the operator at call time (`between` needs a 2-tuple, `in` needs an array, scalar operators reject arrays, unary operators reject values) ‚Äî a mismatch throws synchronously.

### Convenience methods

Every operator has a named helper that delegates to `where`:

```ts
filterQuery()
  .equals('status', 'active')
  .notEquals('role', 'banned')
  .contains('name', 'fleet')
  .in('id', [1, 2, 3])
  .notIn('role', ['banned'])
  .between('age', 18, 65)
  .gt('score', 10).gte('score', 10).lt('score', 90).lte('score', 90)
  .startsWith('code', 'AB')
  .endsWith('email', '.com')
  .isNull('deletedAt').isNotNull('email')
  .isEmpty('bio').isNotEmpty('bio')
```

### `add` ‚Äî accumulating range filters

`where` replaces; `add` accumulates. It is restricted to range operators (`gt`/`gte`/`lt`/`lte`) so you can put two bounds on one field:

```ts
filterQuery()
  .add('createdAt', 'gte', '2026-01-01')
  .add('createdAt', 'lte', '2026-12-31')
// both survive ‚Äî a between-style range on createdAt

// shorthands:
filterQuery().addGte('createdAt', '2026-01-01').addLte('createdAt', '2026-12-31')
```

Calling `add` with a non-range operator throws ‚Äî use `where` (or `between`) for those.

### `remove` and `clear`

```ts
filterQuery()
  .equals('status', 'active')
  .contains('name', 'fleet')
  .remove('status')   // drops all filters for `status`
  .build()
// ‚Üí { filter: { where: [{ field: 'name', operator: 'contains', value: 'fleet' }] } }

filterQuery().equals('a', 1).clear() // resets everything to empty
```

## AND / OR groups

`.or()` and `.and()` take a callback that receives a sub-builder; its conditions become a nested group:

```ts
filterQuery()
  .equals('status', 'active')
  .or((q) => q
    .contains('name', 'sync')
    .contains('email', 'sync'),
  )
  .build()
// ‚Üí status = 'active' AND (name contains 'sync' OR email contains 'sync')
```

<Callout type="info">
Groups are emitted as a `ColumnFilter` with an empty `field` and an `OR`/`AND` array ‚Äî the "group node" shape the server's validator recognizes. Groups are not representable as flat objects, so `toQueryString()` falls back to the indexed `where[i][...]` notation when any group is present; `toFlatObject()` ignores them.
</Callout>

## Sort, pagination, search, include, distinct

```ts
filterQuery()
  .sort('createdAt', 'desc')   // or .sortDesc('createdAt') / .sortAsc('name')
  .search('fleet')             // global term
  .include('role', 'posts')    // relations to eager-load (deduped)
  .distinct('status')          // distinct projection (deduped)
  .page(0, 25)                 // 0-based page index, size (default 25)
  .set('extraKey', 'value')    // arbitrary extra key in the output
```

<Callout type="warn">
`.page(page, size)` uses a **0-based** page index (TanStack convention). The server's `applyFilter` resolves a **1-based** page for Lucid's `paginate()`. If you send the client's page straight through, add `1` at the boundary ‚Äî or use `toQueryString()`, where the page is passed verbatim and you normalize server-side.
</Callout>

## Output formats

| Terminator | Returns | Use for |
|---|---|---|
| `toQueryString()` | a `&`-joined query string | `GET` endpoint URLs |
| `build()` | a `FilterQueryResult` object ‚Äî `{ filter: { where }, sort, paginate, include, search, distinct }` | `POST` search bodies |
| `toFlatObject()` | a flat `{ field: value \| { op: value } }` map | the `filter` bag of a flat request; **drops groups** |

```ts
const builder = filterQuery().gte('age', 18).in('status', ['A', 'B'])

builder.toQueryString() // filter[age][gte]=18&filter[status][]=A&filter[status][]=B
builder.build()         // { filter: { where: [ {age gte 18}, {status in [A,B]} ] } }
builder.toFlatObject()  // { age: { gte: 18 }, status: ['A', 'B'] }
```

## The typed builder

`filterQueryTyped<Fields, Map>()` is the same builder at runtime, but its method signatures restrict field names to a known union ‚Äî and, when you supply a field-type map, restrict each field's operators and value types too. Zero runtime overhead; the typing is the only difference.

### Field names only

```ts
import { filterQueryTyped } from '@agora/filter-client'

type UserFields = 'name' | 'age' | 'status'

const q = filterQueryTyped<UserFields>()
  .contains('name', 'Al')
  .gte('age', 18)
  .sortDesc('age')
  .page(0, 25)
  .build()

// ‚ùå Compile error ‚Äî 'invalid' is not a UserField:
// filterQueryTyped<UserFields>().where('invalid', 'foo')
```

### Field names + types

Pass a second type argument mapping each field to its TS value type. The builder then derives the legal operators and value types per field:

```ts
type UserMap = {
  name: string
  age: number
  status: 'active' | 'inactive'
  createdAt: Date
  deletedAt: Date | null
}

const q = filterQueryTyped<keyof UserMap & string, UserMap>()
  .contains('name', 'Al')        // ‚úÖ string field ‚Üí string operators
  .gte('age', 18)                // ‚úÖ number field ‚Üí ordering operators
  .equals('status', 'active')    // ‚úÖ only the enum members are accepted
  .between('createdAt', d1, d2)  // ‚úÖ Date field ‚Üí ordering + tuple
  .isNull('deletedAt')           // ‚úÖ isNull/isNotNull valid for every field

// ‚ùå contains() is gated to string fields ‚Äî 'age' is a number:
// filterQueryTyped<keyof UserMap & string, UserMap>().contains('age', 'x')

// ‚ùå gte() is gated to orderable fields ‚Äî 'status' is a string enum:
// filterQueryTyped<keyof UserMap & string, UserMap>().gte('status', 'x')
```

The operator matrix the types enforce mirrors the runtime validator exactly:

| Field base type | Allowed operators |
|---|---|
| `string` (incl. string enums) | equality, string ops, array ops, `isEmpty`/`isNotEmpty`, `isNull`/`isNotNull`, `exists`/`notExists` |
| `number` | equality, ordering, `between`/`notBetween`, array ops, null/exists unary |
| `boolean` | equality, array ops, null/exists unary |
| `Date` | equality, ordering, `between`/`notBetween`, array ops, null/exists unary |
| `json` / other / `unknown` | permissive ‚Äî the full operator union |

<Callout type="info">
Nullable fields (`Date | null`) get the **base** type's operators ‚Äî `NonNullable` strips the `null` so `deletedAt: Date | null` still gets ordering operators, plus the always-valid `isNull`/`isNotNull`.
</Callout>

### `whereDynamic` / `sortDynamic` ‚Äî the runtime escape hatch

The typed builder narrows `where` and `sort` to the *known* field union ‚Äî which is exactly what you want until the field name isn't known at compile time: an AG-Grid column id, a user-built query, a field the codegen'd builder doesn't cover. For those, `whereDynamic` and `sortDynamic` take a plain `string` field and deliberately step outside the type-level allow-list.

```ts
// The field comes from runtime state (a grid column, a saved view, user input):
builder.whereDynamic(column.id, filterModel.operator, filterModel.value)

// Unary operators accept the 2-arg form:
builder.whereDynamic('deletedAt', 'isNull')

// Sort counterpart:
builder.sortDynamic(column.id, sortModel.direction) // direction defaults to 'asc'
```

They are **not** unvalidated: `whereDynamic` still checks the operator against the known operator set (an unknown operator throws) and validates the value shape against it, and it keeps the same **replace** semantics as `where` (one filter per field). `sortDynamic` is identical to `sort` at runtime ‚Äî it exists purely so the typed builder can keep `sort` narrowed to the generated union while still offering an explicit untyped entry point.

```ts
import { filterQueryTyped } from '@agora/filter-client'

type UserFields = 'name' | 'age'

filterQueryTyped<UserFields>()
  .contains('name', 'Al')                       // ‚úÖ typed, narrowed field
  .whereDynamic('customAttr.color', 'equals', 'red') // ‚úÖ field outside the union
  .sortDynamic('customAttr.color', 'desc')
```

<Callout type="warn" title="The server allow-list is still the real boundary">
`whereDynamic`/`sortDynamic` only bypass the **client's** compile-time field union ‚Äî a convenience, never a security control. The server's `applyFilter` allow-list is the actual boundary: a dynamic field the server doesn't whitelist is dropped (or rejected under `throwOnInvalid`) regardless of what the client sent.
</Callout>

## Reactivity

The builder is also an observable store, so framework adapters can re-render on mutation. Every mutating method bumps an internal version, invalidates a cached snapshot, and notifies subscribers.

| Member | Purpose |
|---|---|
| `subscribe(listener)` | register a change listener; returns an unsubscribe fn (bound ‚Äî pass it directly) |
| `getSnapshot()` | the current `build()` result, cached with a **stable reference** until the next mutation |
| `getVersion()` | monotonic mutation counter (cheap dependency key) |

This is the exact contract React's `useSyncExternalStore` needs:

<Tabs items={['React', 'Vanilla']}>
  <Tab value="React">
```ts
import { useSyncExternalStore, useRef } from 'react'
import { filterQuery } from '@agora/filter-client'

function useFilterQuery() {
  const ref = useRef(filterQuery())
  const builder = ref.current
  const snapshot = useSyncExternalStore(builder.subscribe, builder.getSnapshot)
  return [builder, snapshot] as const
}

// usage: builder.contains('name', e.target.value) re-renders with the new snapshot
```
  </Tab>
  <Tab value="Vanilla">
```ts
import { filterQuery } from '@agora/filter-client'

const builder = filterQuery()
const unsubscribe = builder.subscribe(() => {
  refetch(builder.toQueryString())
})

builder.equals('status', 'active') // triggers the listener
// ...later
unsubscribe()
```
  </Tab>
</Tabs>

<Callout type="info">
`getSnapshot()` returns the **same object reference** until the next mutation ‚Äî required by `useSyncExternalStore` to avoid infinite render loops. Don't mutate the returned snapshot; treat it as immutable.
</Callout>

## TanStack Table sync

A types-only peer (`@tanstack/table-core`) powers an optional adapter at the `@agora/filter-client/tanstack` subpath. It maps vanilla TanStack Table state ‚Äî column filters, sorting, pagination ‚Äî onto a builder.

```ts
import { filterQuery } from '@agora/filter-client'
import { applyTanstackTableState } from '@agora/filter-client/tanstack'

const body = applyTanstackTableState(filterQuery(), {
  columnFilters: table.getState().columnFilters, // { id, value }[]
  sorting: table.getState().sorting,             // { id, desc }[]
  pagination: table.getState().pagination,       // { pageIndex, pageSize }
  resolveOperator: (id) => (id === 'createdAt' ? 'gte' : 'iContains'),
  fields: ['name', 'status', 'createdAt'],       // optional allowlist; others dropped
})
  .include('author')
  .build()
```

TanStack column filters carry no operator (it lives in the column's `filterFn`), so `resolveOperator(columnId, value)` is the seam where you decide one. The default: array ‚Üí `in`, string ‚Üí `iContains` (matching TanStack's `auto` substring filtering), everything else ‚Üí `equals`.

For a one-shot build straight from state:

```ts
import { tanstackTableToFilterQuery } from '@agora/filter-client/tanstack'

const body = tanstackTableToFilterQuery({
  columnFilters: table.getState().columnFilters,
  sorting: table.getState().sorting,
  pagination: table.getState().pagination,
})
```

<Callout type="warn">
TanStack's `pagination.pageIndex` is 0-based and is passed straight to the builder's `.page()`. Keep the [0-based vs 1-based](#sort-pagination-search-include-distinct) note in mind when the server reads it.
</Callout>

## Round trip

The client and server agree on the wire format by construction ‚Äî what `.toQueryString()` emits is what `parseFilterRequest` reads:

```ts
// client
const qs = filterQuery().contains('name', 'Al').gte('age', 18).sortDesc('createdAt').toQueryString()

// server
const input = parseFilterRequest(request.qs())
applyFilter(User.query(), input, { allowed: ['name', 'age'], sortable: ['createdAt'] })
```

See [Operators](/docs/filter/guides/operators) for the full wire-format table and [Getting Started](/docs/filter/getting-started) for the end-to-end flow.
