`@adonis-agora/filter` drives a Lucid query builder through a structural interface (`QueryBuilderLike`). It does not invent its own join machinery — instead it composes with Lucid's relationship APIs (`preload`, `whereHas`, `withCount`). This guide shows the two ways relations show up in a filtered endpoint: **filtering the related rows you load**, and **filtering the parent rows by a relation**.

<Callout type="info">
The structural adapter means anything that satisfies `QueryBuilderLike` — a model query, a relation sub-query inside a `whereHas`/`preload` callback — can be handed to `applyFilter` (or to the lower-level `applyColumnFilters` / `applySort` / `applySearch`). Relations are just nested query builders.
</Callout>

---

## Filtering a preloaded relation

`preload`'s callback receives a relation query builder, which is `QueryBuilderLike`. Run a separate parsed input through it to filter the related rows you eager-load — without touching which parents come back:

```ts title="app/controllers/users_controller.ts"
import { parseFilterRequest, applyFilter } from '@adonis-agora/filter'
import User from '#models/user'

async index({ request }: HttpContext) {
  // parent filters live under `filter[...]`; relation filters under their own param
  const userInput = parseFilterRequest(request.qs())
  const postInput = parseFilterRequest({ filter: request.qs().postFilter })

  const query = User.query().preload('posts', (postsQuery) => {
    applyFilter(postsQuery, postInput, {
      allowed: ['title', 'status'],
      sortable: ['createdAt'],
    })
  })

  const { page, size } = applyFilter(query, userInput, {
    allowed: ['name', 'email', 'status'],
  })

  return query.paginate(page, size)
}
```

Each `applyFilter` enforces its own allow-list, so the relation's filterable columns are scoped independently of the parent's.

<Callout type="warn">
`applyFilter` returns pagination, but you almost never want to paginate inside a `preload` callback — paginating a to-many relation per parent row corrupts counts. Ignore the returned `{ page, size }` there (or call the lower-level `applyColumnFilters` / `applySort` directly).
</Callout>

## Filtering the parent by a relation (`whereHas`)

To restrict which *parents* come back based on their related rows, apply the filter inside a `whereHas` callback. The callback's sub-query is also `QueryBuilderLike`:

```ts
const query = User.query().whereHas('posts', (postsQuery) => {
  applyColumnFilters(postsQuery, [
    { field: 'status', operator: 'equals', value: 'published' },
    { field: 'title', operator: 'contains', value: 'release' },
  ])
})
// → users having at least one post WHERE status = 'published' AND title ILIKE '%release%'
```

`applyColumnFilters`, `applySort`, and `applySearch` are exported from `@adonis-agora/filter` for exactly this — applying a piece of the filter pipeline to an arbitrary sub-builder without the allow-list/pagination wrapper.

```ts
import { applyColumnFilters, applySearch, applySort } from '@adonis-agora/filter'
```

<Callout type="info">
These low-level functions do **not** allow-list. When you feed them client-derived `ColumnFilter[]`, prune first (e.g. run the input through `applyFilter` against a scratch builder, or validate the fields yourself) so a `whereHas` sub-query can't reference arbitrary relation columns.
</Callout>

## Dotted field names are relation paths

Field names may contain dots — `validateColumnFilter` accepts `^[a-zA-Z_][a-zA-Z0-9_.]*$`. A dotted field like `author.name` is a **relation path**: every segment but the last is a relation hop, and the adapter translates it into a nested Lucid `whereHas` subquery (the final segment is the bare column the operator lands on inside it):

```ts
applyFilter(query, input, { allowed: ['author.name', 'status'] })
// ?filter[author.name]=Al  →  whereHas('author', (q) => q.where('name', 'Al'))
```

So you do **not** need to join the relation yourself — Lucid builds the subquery. Deeper paths nest `whereHas` within `whereHas` (`posts.comments.body` → `whereHas('posts', q => q.whereHas('comments', c => c.where('body', …)))`).

<Callout type="info">
The declarative way to whitelist relation paths (with a depth cap) is a `defineFilter` spec's `relations` map — see [Relation Filtering](/docs/filter/definitions/relations). With a raw `FilterConfig` you allow-list the dotted path by name (`allowed: ['author.name']`) as above.
</Callout>

<Callout type="warn" title="whereHas, not a joined column">
Because a dotted filter becomes a `whereHas` subquery, the relation does **not** need to be joined for filtering. If you have genuinely joined a relation and want to reference its column directly (a real `where('authors.name', …)` on the joined table) rather than a subquery, filter on a non-dotted alias you select for it, or use the low-level `applyColumnFilters` on the joined builder.
</Callout>

## Counting and ordering by a relation

Because the parent query is just a Lucid query builder, you can layer relation aggregates before or after `applyFilter`:

```ts
const query = User.query().withCount('posts')

const { page, size } = applyFilter(query, input, {
  allowed: ['name', 'status'],
  sortable: ['name', 'createdAt', 'posts_count'], // allow ordering by the aggregate alias
})

return query.paginate(page, size)
```

Since `posts_count` is selected by `withCount`, listing it in `sortable` lets clients sort by it via `?sort=-posts_count`.
